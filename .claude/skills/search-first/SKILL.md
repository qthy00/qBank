# /search-first — 编码前先研究

系统化"在实现之前搜索现有解决方案"的工作流。

## 触发

在以下情况下使用此技能：
- 开始一个可能有现有解决方案的新功能
- 添加依赖项或集成
- 用户要求"添加 X 功能"而你即将编写代码
- 在创建新工具、辅助函数或抽象之前

## 工作流

```
┌─────────────────────────────────────────────┐
│  1. 需求分析                                │
│     定义需要什么功能                         │
│     识别语言/框架约束                        │
├─────────────────────────────────────────────┤
│  2. 并行搜索（研究代理）                      │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │  npm /   │ │  MCP /   │ │  GitHub / │  │
│     │  PyPI    │ │  Skills  │ │  Web      │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 评估                                    │
│     为候选者评分（功能、维护、               │
│     社区、文档、许可证、依赖）                │
├─────────────────────────────────────────────┤
│  4. 决策                                    │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │  采用   │  │  扩展    │  │  构建   │  │
│     │  原样   │  │  /包装   │  │  自定义 │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 实现                                    │
│     安装包 / 配置 MCP /                      │
│     编写最小自定义代码                       │
└─────────────────────────────────────────────┘
```

## 决策矩阵

| 信号 | 操作 |
|------|------|
| 完全匹配、维护良好、MIT/Apache | **采用** — 直接安装使用 |
| 部分匹配、基础良好 | **扩展** — 安装 + 编写薄包装 |
| 多个弱匹配 | **组合** — 组合 2-3 个小包 |
| 没有找到合适的 | **构建** — 编写自定义，但基于研究 |

## 如何使用

### 快速模式（内联）

在编写工具或添加功能之前，在脑海中运行：

0. 这已经在仓库中存在了吗？→ 先通过 `rg` 搜索相关模块/测试
1. 这是常见问题吗？→ 搜索 npm/PyPI
2. 有为此的 MCP 吗？→ 检查 `~/.claude/settings.json` 并搜索
3. 有为此的技能吗？→ 检查 `~/.claude/skills/`
4. 有 GitHub 实现/模板吗？→ 在编写全新代码之前运行 GitHub 代码搜索

### 完整模式（代理）

对于非平凡功能，启动研究代理：

```
Task(subagent_type="general-purpose", prompt="
  研究以下内容的现有工具：[描述]
  语言/框架：[LANG]
  约束：[ANY]

  搜索：npm/PyPI、MCP 服务器、Claude Code 技能、GitHub
  返回：带推荐的结构化比较
")
```

## 按类别搜索快捷方式

### 开发工具
- Linting → `eslint`、`ruff`、`textlint`、`markdownlint`
- 格式化 → `prettier`、`black`、`gofmt`
- 测试 → `jest`、`pytest`、`go test`
- 提交前 → `husky`、`lint-staged`、`pre-commit`

### AI/LLM 集成
- Claude SDK → Context7 获取最新文档
- 提示管理 → 检查 MCP 服务器
- 文档处理 → `unstructured`、`pdfplumber`、`mammoth`

### 数据与 API
- HTTP 客户端 → `httpx` (Python)、`ky`/`got` (Node)
- 验证 → `zod` (TS)、`pydantic` (Python)
- 数据库 → 首先检查 MCP 服务器

### 内容与发布
- Markdown 处理 → `remark`、`unified`、`markdown-it`
- 图像优化 → `sharp`、`imagemin`

## 集成点

### 与 planner 代理
Planner 应在第 1 阶段（架构审查）之前调用 researcher：
- Researcher 识别可用工具
- Planner 将它们纳入实施计划
- 避免计划中"重新发明轮子"

### 与 architect 代理
Architect 应就以下方面咨询 researcher：
- 技术栈决策
- 集成模式发现
- 现有参考架构

### 与 iterative-retrieval 技能
结合进行渐进式发现：
- 第 1 轮：广泛搜索（npm、PyPI、MCP）
- 第 2 轮：详细评估顶级候选
- 第 3 轮：测试与项目约束的兼容性

## 示例

### 示例 1："添加死链检查"
```
需求：检查 markdown 文件中的失效链接
搜索：npm "markdown dead link checker"
找到：textlint-rule-no-dead-link（评分：9/10）
操作：采用 — npm install textlint-rule-no-dead-link
结果：零自定义代码，久经考验的解决方案
```

### 示例 2："添加 HTTP 客户端包装器"
```
需求：具有重试和超时处理的弹性 HTTP 客户端
搜索：npm "http client retry"、PyPI "httpx retry"
找到：got (Node) 带重试插件、httpx (Python) 内置重试
操作：采用 — 直接使用 got/httpx 配合重试配置
结果：零自定义代码，久经生产验证的库
```

### 示例 3："添加配置文件 linter"
```
需求：根据模式验证项目配置文件
搜索：npm "config linter schema"、"json schema validator cli"
找到：ajv-cli（评分：8/10）
操作：采用 + 扩展 — 安装 ajv-cli，编写项目特定模式
结果：1 个包 + 1 个模式文件，无自定义验证逻辑
```

## 反模式

- **直接跳到代码**：不检查是否存在就编写工具
- **忽略 MCP**：不检查 MCP 服务器是否已提供该功能
- **过度定制**：过度包装库，失去其优势
- **依赖膨胀**：为一个小功能安装庞大的包
