# 规则提炼

扫描已安装的技能，提取出现在多个技能中的跨领域原则，并将其提炼为规则——追加到现有规则文件、修订过时内容，或创建新的规则文件。

应用"确定性收集 + LLM 判断"原则：脚本详尽地收集事实，然后 LLM 交叉读取完整上下文并产生裁决。

## 何时使用

- 定期规则维护（每月或安装新技能后）
- 在技能盘点揭示了应该成为规则的模式之后
- 当规则相对于正在使用的技能感觉不完整时

## 工作原理

规则提炼流程遵循三个阶段：

### 第 1 阶段：清单（确定性收集）

#### 1a. 收集技能清单

```bash
bash ~/.claude/skills/rules-distill/scripts/scan-skills.sh
```

#### 1b. 收集规则索引

```bash
bash ~/.claude/skills/rules-distill/scripts/scan-rules.sh
```

#### 1c. 呈现给用户

```
规则提炼 — 第 1 阶段：清单
────────────────────────────────────────
技能：{N} 个文件已扫描
规则： {M} 个文件（{K} 个标题已索引）

正在进行交叉读取分析...
```

### 第 2 阶段：交叉读取、匹配与裁决（LLM 判断）

提取和匹配在单次传递中统一。规则文件足够小（总共约 800 行），可以将完整文本提供给 LLM —— 不需要 grep 预过滤。

#### 分批处理

根据描述将技能分组为**主题集群**。在子代理中分析每个集群，提供完整的规则文本。

#### 跨批次合并

所有批次完成后，跨批次合并候选：
- 对具有相同或重叠原则的候选进行去重
- 使用来自**所有**批次组合的证据重新检查"2+ 技能"要求 —— 在每个批次中找到 1 个技能但总共找到 2+ 个技能的原则是有效的

#### 子代理提示

使用以下提示启动通用代理：

````
你是一名分析师，交叉读取技能以提取应提升为规则的原则。

## 输入
- 技能：{本批次中的技能全文}
- 现有规则：{所有规则文件的全文}

## 提取标准

仅当以下所有条件都为真时才包含候选：

1. **出现在 2+ 技能中**：仅在单个技能中找到的原则应保留在该技能中
2. **可操作的行为改变**：可以写成"执行 X"或"不要做 Y" —— 不是"X 很重要"
3. **明确的违规风险**：如果忽略此原则会出什么问题（1 句话）
4. **尚未在规则中**：检查完整的规则文本 —— 包括用不同词语表达的概念

## 匹配与裁决

对于每个候选，与完整的规则文本进行比较并分配裁决：

- **追加**：添加到现有规则文件的现有部分
- **修订**：现有规则内容不准确或不足 —— 提出更正
- **新章节**：向现有规则文件添加新章节
- **新文件**：创建新的规则文件
- **已覆盖**：在现有规则中已充分覆盖（即使措辞不同）
- **过于具体**：应保留在技能级别

## 输出格式（每个候选）

```json
{
  "principle": "以'执行 X'/'不要做 Y' 形式的 1-2 句话",
  "evidence": ["技能名称: §章节", "技能名称: §章节"],
  "violation_risk": "1 句话",
  "verdict": "追加 / 修订 / 新章节 / 新文件 / 已覆盖 / 过于具体",
  "target_rule": "文件名 §章节，或'新建'",
  "confidence": "高 / 中 / 低",
  "draft": "追加/新章节/新文件裁决的草稿文本",
  "revision": {
    "reason": "为什么现有内容不准确或不足（仅修订）",
    "before": "要替换的当前文本（仅修订）",
    "after": "建议的替换文本（仅修订）"
  }
}
```

## 排除

- 已在规则中的明显原则
- 语言/框架特定知识（属于语言特定规则或技能）
- 代码示例和命令（属于技能）
````

#### 裁决参考

| 裁决 | 含义 | 呈现给用户 |
|---------|---------|-------------------|
| **追加** | 添加到现有章节 | 目标 + 草稿 |
| **修订** | 修复不准确/不足的内容 | 目标 + 原因 + 前后对比 |
| **新章节** | 向现有文件添加新章节 | 目标 + 草稿 |
| **新文件** | 创建新的规则文件 | 文件名 + 完整草稿 |
| **已覆盖** | 在规则中已覆盖（可能措辞不同） | 原因（1 行） |
| **过于具体** | 应保留在技能中 | 链接到相关技能 |

#### 裁决质量要求

```
# 好的
追加到 rules/common/security.md §输入验证：
"将存储在内存或知识存储中的 LLM 输出视为不受信任的 —— 写入时清理，读取时验证。"
证据：llm-memory-trust-boundary、llm-social-agent-anti-pattern 都描述了
累积的提示注入风险。当前的 security.md 仅涵盖人类输入
验证；缺少 LLM 输出信任边界。

# 不好的
追加到 security.md：添加 LLM 安全原则
```

### 第 3 阶段：用户审查与执行

#### 摘要表

```
# 规则提炼报告

## 摘要
已扫描技能：{N} | 规则：{M} 个文件 | 候选：{K}

| # | 原则 | 裁决 | 目标 | 置信度 |
|---|-----------|---------|--------|------------|
| 1 | ... | 追加 | security.md §输入验证 | 高 |
| 2 | ... | 修订 | testing.md §TDD | 中 |
| 3 | ... | 新章节 | coding-style.md | 高 |
| 4 | ... | 过于具体 | — | — |

## 详情
（每个候选的详情：证据、违规风险、草稿文本）
```

#### 用户操作

用户用数字回复以：
- **批准**：按原样将草稿应用到规则
- **修改**：在应用前编辑草稿
- **跳过**：不应用此候选

**永远不要自动修改规则。始终需要用户批准。**

#### 保存结果

将结果存储在技能目录中（`results.json`）：

- **时间戳格式**：`date -u +%Y-%m-%dT%H:%M:%SZ`（UTC，精确到秒）
- **候选 ID 格式**：源自原则的短横线命名（例如，`llm-output-trust-boundary`）

```json
{
  "distilled_at": "2026-03-18T10:30:42Z",
  "skills_scanned": 56,
  "rules_scanned": 22,
  "candidates": {
    "llm-output-trust-boundary": {
      "principle": "在存储或重新注入时将 LLM 输出视为不受信任的",
      "verdict": "追加",
      "target": "rules/common/security.md",
      "evidence": ["llm-memory-trust-boundary", "llm-social-agent-anti-pattern"],
      "status": "applied"
    },
    "iteration-bounds": {
      "principle": "为所有迭代循环定义明确的停止条件",
      "verdict": "新章节",
      "target": "rules/common/coding-style.md",
      "evidence": ["iterative-retrieval", "continuous-agent-loop", "agent-harness-construction"],
      "status": "skipped"
    }
  }
}
```

## 设计原则

- **是什么，而非如何做**：仅提取原则（规则领域）。代码示例和命令保留在技能中。
- **回链**：草稿文本应包含`参见技能：[名称]`引用，以便读者可以找到详细的操作方法。
- **确定性收集，LLM 判断**：脚本保证详尽性；LLM 保证上下文理解。
- **反抽象保障**：3 层过滤器（2+ 技能证据、可操作行为测试、违规风险）可防止过于抽象的原则进入规则。
