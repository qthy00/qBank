# 任务跟踪技能

## 何时使用

✅ **适用场景**
- 用户明确要求跟踪任务（"创建任务跟踪"、"记录进度"、"跟踪这个任务"）
- 复杂的多步骤开发任务
- 需要跨会话恢复的工作
- 需要记录方案设计和决策过程
- 需要跟踪问题和解决方案

❌ **不适用场景**
- 简单的单步操作
- 纯咨询性对话

---

## 使用流程

### 步骤1：创建任务文档
当用户需要跟踪新任务时，按以下步骤执行：

1. 确定任务标题和简要描述
2. 创建任务文档到 `docs/task/active/` 目录
3. 使用命名格式：`task-{日期}-{任务标识}.md`
4. 填充基础信息（状态、时间、Git分支）

### 步骤2：更新任务进度
每次会话或每次进度更新时：

1. 更新**实现步骤**中的复选框状态
2. 更新**当前进度**章节
3. 如有问题，添加到**问题记录**
4. 如有决策，记录到**关键决策**
5. 更新时间戳

### 步骤3：完成任务归档
任务完成后：

1. 将所有状态改为"已完成"
2. 更新完成时间
3. 将文件移动到 `docs/task/archive/{年月}/`

## 最佳实践

### 1. 任务粒度控制
- 单个任务应在 **1天** 内可完成
- 超过 **1天** 的任务应拆分为子任务
- 每个子任务应有明确的交付物

### 2. 进度更新频率
- **每日** 更新进度和遇到的问题
- **每次** 会话结束时更新状态
- **关键节点**（方案确定、技术选型、问题修复）立即记录

### 3. 问题记录规范
- **及时记录**：发现问题立即记录
- **详细描述**：包含环境、复现步骤、错误信息
- **跟踪状态**：明确问题状态（待解决/进行中/已解决）
- **记录方案**：包含尝试过的解决方案

### 4. 决策记录要求
- **明确内容**：决策的具体内容
- **记录时间**：决策发生的时间
- **说明理由**：为什么做此决策
- **记录权衡**：考虑过的替代方案

### 5. 文档维护
- **定期清理**：已完成的任务及时归档
- **链接管理**：确保相关文件链接有效
- **版本控制**：重要变更记录历史记录

## 任务状态流转

```
创建任务 → 进行中 → 已完成
            ↓
         已暂停 → 进行中
            ↓
         已取消
```

| 状态 | 说明 | 可转换到 |
| --- | --- | --- |
| **进行中** | 任务正在执行 | 已完成、已暂停、已取消 |
| **已完成** | 任务已完成并验收 | - |
| **已暂停** | 任务暂时中断 | 进行中、已取消 |
| **已取消** | 任务不再执行 | - |
