# /next - 下一步建议

作为项目开发顾问，综合分析项目全貌后，推荐**一个最优下一步**和备选方向。

---

## 第一步：收集项目全貌（5个数据源，并行执行）

### 1.1 Git 提交分析（过滤非开发提交）

```bash
# 获取最近 10条提交，包含 author/committer/message 信息
git log -10 --format="%H|%an|%cn|%s" --no-merges
```

**过滤规则** （以下提交不作为"最近在做什么"的判断依据）：

| 过滤条件               | 识别方式                                            | 分类              |
|:-------------------|:------------------------------------------------|:----------------|
| Cherry-pick 提交     | commit message 包含 `cherry picked from`          | 同步提交            |
| 上游同步提交             | commit message 以 `sync:` 或 `upstream:` 开头       | 同步提交            |
| Author ≠ Committer | author 和 committer 名字不同                         | 可能是 cherry-pick |
| 文档/配置提交            | commit message 以 `docs:` `chore:` `style:`   开头 | 非功能提交           |

**输出**：
- 开发提交列表（排除同步/cherry-pick/docs 后的真正开发活动）
- 同步提交列表（单独标注：不影响建议方向）
- 最近开发方向判断（根据开发提交的scope和内容）

### 1.2 活跃任务文档（最重要的数据源）

```bash
# 扫描活跃任务
ls docs/tasks/active/ 2>/dev/null || echo "目录不存在"
```

对每个活跃任务文档，**必须深度读取以下区域**：

| 读取区域        | 提取信息                          | 用途           |
|:------------|:------------------------------|:-------------|
| 文件头部（状态/时间） | 状态（进行中/暂停）、最后更新时间             | 判断是否停滞       |
| 实现步骤        | `- [ ]` 和 `- [x]` 统计，下一个未完成步骤 | 计算进度、确定下一步   |
| 当前进度        | "正在进行"、"下一步操作"、"阻塞问题"         | **直接作为推荐依据** |
| 问题记录        | 状态为"待解决" 或 "进行中"的问题           | 判断是否有阻塞      |
| 关键决策        | 技术选型、架构模式                     | 理解任务上下文      |

**关键判断**：
- 如果任务有"阻塞问题"（"待解决"） -> 推荐解决阻塞
- 如果任务 "下一步操作" 明确 -> **直接推荐任务文档中的下一步**
- 如果任务超过7天无更新 -> 提醒用户是否需要继续或暂停

### 1.3 待办清单

```bash
# 读取待办清单
cat docs/待办清单.md 2>/dev/null || echo "文件不存在"
```

提取：
- 高优先级待办列表
- 中优先级待办列表
- 本周目标
- 截止日期临近的任务

### 1.4 项目状态

```bash
# 读取项目状态
cat docs/项目状态.md 2>/dev/null || echo "文件不存在"
```

提取：
- 当前重点
- 下一步计划
- 模块进度
- 里程碑状态（即将到期的）
- 问题与风险

### 1.5 需求文档

```bash
# 读取需求文档
cat docs/需求文档.md 2>/dev/null || echo "文件不存在"
```

提取：
- 未实现的功能需求（REQ-xxx 状态）
- 优先级高的需求
- 前置依赖已满足的需求

## 第二步：分析决策因素

### 2.1 阻塞检查（最高优先级）

检查是否有阻塞因素：

| 阻塞类型 | 检测方式 | 建议 |
|:---------|:---------|:-----|
| 活跃任务有阻塞问题 | 任务文档中"问题记录"有"待解决" | **必须优先解决阻塞** |
| 前置依赖未完成 | 需求文档中REQ依赖未满足 | 等待或调整计划 |
| 资源不足 | 代码/设计/接口缺失 | 协调资源或调整范围 |
| 技术方案待确认 | 任务文档中"关键决策"未确定 | 组织讨论确定方案 |

**如果存在阻塞** -> 推荐解决阻塞（不继续后续分析）

### 2.2 活跃任务检查（高优先级）

检查活跃任务状态：

| 任务状态 | 处理方式 |
|:---------|:---------|
| 有进行中任务，下一步明确 | **推荐该任务的下一步** |
| 有进行中任务，下一步不明确 | 帮助分析并明确下一步 |
| 多个进行中任务 | 推荐优先级最高的任务 |
| 所有任务暂停/完成 | 推荐开始新任务 |

### 2.3 待办清单检查（中优先级）

如果没有活跃任务或活跃任务不明确：

| 待办情况 | 处理方式 |
|:---------|:---------|
| 有高优先级待办 | 推荐最高优先级待办 |
| 有高优先级待办且截止临近 | **推荐该待办（加急）** |
| 无高优先级待办，有中规级 | 推荐中优先级待办 |
| 待办清单为空 | 推荐需求文档中的下一个功能 |

### 2.4 里程碑检查

检查里程碑时间：

| 里程碑状态 | 处理方式 |
|:---------|:---------|
| 有里程碑3天内到期 | **紧急：推荐完成该里程碑相关任务** |
| 有里程碑7天内到期 | 推荐加速该里程碑相关任务 |
| 无紧急里程碑 | 按正常优先级推荐 |

### 2.5 依赖关系分析

分析任务/需求的依赖关系：

```
如果 任务B 依赖 任务A:
- 任务A 未完成 -> 推荐先完成任务A
- 任务A 已完成 -> 可以开始任务B
```

## 第四步：特殊场景处理

### 场景1：首次使用（无文档无任务）

```
🎯 推荐下一步

行动: 初始化项目文档

原因:
1. 检测到项目文档尚未创建
2. 建立文档是项目管理的基础

操作:
1. 运行 /init-docs 初始化项目文档
2. 或运行头脑风暴讨论功能方案

预期产出:
- 项目状态.md
- 待办清单.md
- 需求文档.md
```

### 场景2：有活跃任务但有阻塞

```
🎯 推荐下一步

行动: 解决"订单支付功能"的阻塞问题

来自: 任务文档 docs/tasks/active/订单支付.md

阻塞问题: 支付宝SDK申请审核中

建议解决方案:
1. 联系支付宝客服加速审核
2. 使用沙箱环境先行开发
3. 考虑接入备用支付方式

原因:
1. 该阻塞导致整个支付功能无法继续
2. 影响后续订单流程测试
3. 已阻塞3天，需要尽快解决

预计时间: 1-2天
```

### 场景3：有多个高优先级任务

```
🎯 推荐下一步

行动: 实现用户权限管理模块的RBAC设计

来自: 待办清单 - 高优先级

原因:
1. 权限管理是其他功能的前置依赖
2. 截止日为3天后，时间紧迫
3. 已与前端确认接口需求

备选方案:
- 可先完成权限数据库设计，接口稍后

预计时间: 2天
```

### 场景4：里程碑即将到期

```
🎯 推荐下一步

行动: 完成"v1.0核心功能"里程碑剩余任务

来自: 里程碑计划 (剩余3天)

剩余任务:
- 订单查询接口 (50% -> 需完成)
- 支付回调处理 (0% -> 需开始)

原因:
1. 里程碑3天后到期
2. 这两个功能是v1.0的核心
3. 延期会影响测试计划

建议:
- 集中精力完成这两个功能
- 非核心功能可延期到v1.1

预计时间: 3天（需加班）
```

## 执行流程总结

1. **并行收集数据**
   - Git提交分析（过滤非开发提交）
   - 活跃任务扫描（深度读取）
   - 待办清单读取
   - 项目状态读取
   - 需求文档读取

2. **分析决策因素**
   - 检查阻塞问题（最高优先级）
   - 检查活跃任务状态
   - 检查待办清单优先级
   - 检查里程碑时间
   - 分析依赖关系

3. **生成推荐**
   - 应用决策树
   - 生成最优推荐
   - 提供备选方案
   - 提示风险

4. **执行与跟踪**
   - 确认用户选择
   - 创建活跃任务（可选）
   - 更新待办清单（可选）
   - 提供后续命令建议
