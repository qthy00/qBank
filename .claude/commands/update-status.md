# /update-status - 智能更新项目状态

作为项目状态更新助手，智能更新项目管理文档。支持自动创建缺失文档、联动更新三个文档。

---

## 第一步：智能检测文档状态

### 1.1 检查三个核心文档

检查 `docs/` 目录下是否存在以下三个文件：
- `docs/项目状态.md`
- `docs/待办清单.md`
- `docs/需求文档.md`

### 1.2 自动创建缺失文档

**如果 `docs/项目状态.md` 不存在**：
1. 读取模板 `.claude/templates/项目状态模板.md`
2. 扫描业务模块抹去初始数据（参考 /init-docs 的扫描逻辑）
3. 创建 `docs/项目状态.md`
4. 输出提示：`已自动创建 docs/项目状态.md`

**如果 `docs/待办清单.md` 不存在**：
1. 读取模板 `.claude/templates/待办清单模板.md`
2. 扫描代码中的 TODO/FIXME 作为初始待办
3. 创建 `docs/待办清单.md`
4. 输出提示：`已自动创建 docs/待办清单.md`

**如果 `docs/需求文档.md` 不存在**：
1. 读取模板 `.claude/templates/需求文档模板.md`
2. 从 package.json / pom.xml 中提取技术栈信息
3. 创建 `docs/需求文档.md`
4. 输出提示：`已自动创建 docs/需求文档.md`

## 第二步：分析最新状态

### 2.1 读取现有文档

- 读取 `docs/项目状态.md` 的当前内容
- 记录"已完成"、"进行中"、"待办"各区域的任务
- 提取里程碑状态和进度

### 2.2 分析Git提交（过滤非开发提交）

```bash
# 获取最近 10条提交，包含 author/committer/message 信息
git log -10 --format="%H|%an|%cn|%s" --no-merges
```

**过滤规则** （以下提交不作为"最近在做什么"的判断依据）：

| 过滤条件               | 识别方式                                            | 分类              |
|:-------------------|:------------------------------------------------|:----------------|
| Cherry-pick 提交     | commit message 包含 `cherry picked from`          | 同步提交            |
| 上游同步提交             | commit message 以 `sync:` 或 `upstream:` 开头       | 同步提交            |
| Author ≠ Committer | author 和 committer 名字不同                         | 可能是 cherry-pick |
| 文档/配置提交            | commit message 以 `docs:` `chore:` `style:`   开头 | 非功能提交           |

**输出**：只展示过滤后的真实开发提交

### 2.3 提取开发活动信息

根据提交信息分析：
- 最近开发方向（根据 commit message 关键词）
- 涉及的模块（根据文件路径）
- 开发频率（提交密度）

### 2.4 扫描活跃任务

```bash
ls docs/tasks/active/ 2>/dev/null || echo "无活跃任务目录"
```

对每个活跃任务文档：
- 读取状态和进度
- 统计已完成/未完成步骤
- 检查是否有阻塞问题

### 2.5 扫描代码TODO

```bash
grep -r "TODO\|FIXME" --include="*.java" --include="*.ts" --include="*.vue" --include="*.tsx" . 2>/dev/null | head -20
```

## 第三步：联动更新三个文档

### 3.1 更新项目状态.md

#### 更新整体进度

根据以下数据计算整体进度：
- 活跃任务的平均进度
- 已完成模块数 / 总模块数
- Git提交反映的开发活跃度

```
整体进度 = (已完成模块数 × 100% + 进行中模块进度之和) / 总模块数
```

#### 更新模块进度

根据Git提交涉及的文件路径，判断哪些模块有进展：
- 如果某模块有提交 -> 状态设为"开发中"
- 如果某模块所有功能完成 -> 状态设为"已完成"
- 计算每个模块的进度百分比

#### 更新本周完成

根据Git提交记录，提取本周（最近7天）完成的任务：
- feat: 新功能完成
- fix: Bug修复
- refactor: 重构完成

#### 更新下周计划

根据以下信息推断：
- 活跃任务的下一步操作
- 待办清单的高优先级任务
- 项目里程碑计划

#### 更新问题与风险

检查：
- 活跃任务中的阻塞问题
- 代码中的FIXME注释
- 超过期限的里程碑

### 3.2 联动更新待办清单.md

#### 同步已完成的任务

- 从Git提交中提取已完成的任务
- 将对应待办项标记为 `- [x]` 已完成
- 添加完成时间注释

#### 同步新增的任务

- 从代码TODO中提取新任务
- 从活跃任务的实现步骤中提取
- 根据优先级自动分类

#### 更新统计信息

```
- 总计任务数: ｛N｝
- 高优先级: ｛N｝
- 中优先级: ｛N｝
- 低优先级: ｛N｝
- 已完成: ｛N｝
- 完成率: ｛N｝%
```

### 3.3 联动更新需求文档.md

#### 更新功能实现状态

根据代码实现情况，更新需求文档中的验收标准：
- 已实现的功能 -> 标记 `[x]`
- 部分实现的功能 -> 标记 `[~]` 或保持 `[]` 加注释
- 未实现的功能 -> 保持 `[]`

#### 更新技术栈信息

从 package.json / pom.xml 中提取最新的技术栈版本：
- 检查依赖版本变化
- 更新技术栈表格

## 第五步：可选的交互操作

### 5.1 确认更新（如果变更较大）

如果检测到重大变更（如：删除任务、修改里程碑日期），提示用户确认：

```
检测到以下重大变更，请确认：

1. [删除] 里程碑"测试阶段" (原因：已合并到开发阶段)
2. [修改] 任务"支付接口"优先级 高 -> 中
3. [修改] 截止日期 2024-01-20 -> 2024-01-25

是否确认更新？ (是/否/编辑)
- 是: 应用所有变更
- 否: 放弃本次更新
- 编辑: 逐个确认变更
```

### 5.2 手动调整

允许用户手动调整更新结果：

```
是否需要手动调整更新内容？

1. 调整项目进度
2. 添加/删除任务
3. 修改里程碑
4. 无需调整，完成更新
```

## 执行流程总结

1. **检测文档状态** - 检查三个核心文档是否存在
2. **自动创建缺失** - 为缺失的文档创建初始版本
3. **读取现有数据** - 读取当前项目状态和待办清单
4. **分析Git提交** - 提取最近开发活动（过滤非开发提交）
5. **扫描活跃任务** - 读取 docs/tasks/active/ 下的任务
6. **扫描代码TODO** - 提取代码中的待办事项
7. **计算新状态** - 根据以上数据计算更新后的状态
8. **联动更新文档** - 同时更新三个文档保持一致
9. **生成更新报告** - 输出变更摘要和详情
10. **确认/调整** - 如有重大变更，提示用户确认

---

## 特殊场景处理

### 场景1：首次使用（文档都不存在）
- 自动创建三个文档的初始版本
- 基于代码扫描填充初始数据
- 提示用户：首次初始化完成，建议检查并调整

### 场景2：Git提交为空（新仓库）
- 仅基于现有文档状态更新
- 提示用户：暂无Git提交记录，手动更新状态

### 场景3：活跃任务和文档不一致
- 以活跃任务为准更新项目状态
- 提示用户：检测到任务与文档不一致，已自动同步

### 场景4：代码TODO和待办清单重复
- 自动去重合并
- 保留待办清单中的优先级信息
